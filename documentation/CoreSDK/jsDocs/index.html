cpaasCoreSDK

CPaaS CoreSDK

Purpose

This is a JavaScript SDK library for accessing the T-Mobile Communications Platform as a Service (CPaaS) network. This library abstracts the intricate Client registration processes into high-level functions. The CpaasCoreSDK module provides APIs for client registration onto the CPaaS network. It also provides internal APIs required by other CPaaS SDKs. The CpaasCoreSDK module must be included in any project along with Calling, Messaging, or other SDKs.

Features

The CpaasCoreSDK root interface provides access to the Registration Manager interface along with several other internal interfaces.

The Registration Manager provides essential APIs and Call-back event notifiers needed for a client application to access account and line information from the CPaaS network.

The CpaasCoreSDK library will work with both JavaScript and TypeScript projects, but is only designed for a web application running inside a web browser.

Documentation

All public API methods, common types and interfaces have been documented with JsDocs. Use a web browser to open the 'index.html' file under 'built/jsDocs' directory.

How to use

Copy the cpaasCoreSDK.js and optional cpaasCoreSDK.d.ts files into a local directory within your application project.

Include the cpaasCoreSDK library into your application code, then later instantiate and initialize.

import * as cpaasCoreSdkLib from '../extLibs/cpaasCoreSDK'
...
const cpaasCoreSDK = cpaasCoreSdkLib.CpaasCoreSDK.getInstance()
const regMgr = cpaasCoreSdkLib.getRegistrationManager()
regMgr.initialize(ClientConfig)
regMgr.addConnectionEventListener(ourEventListener_callBack)
regMgr.appStart()

CoreSDK General Theory of Operation

The application code needs to instantiate CoreSDK library prior to instantiating any of the other CPaaS JavaScript SDKs.
A reference to the CoreSDK will be needed to instantiate the other SDKs, which need the CoreSDK for registration, configuration, and CPaaS network API access.
The application code gets the Registration Manager interface from the CoreSDK.
The Registration Manager singleton object provides access to methods and event call-backs which allow the application code to determine the account lines are available, select lines for registration, monitor connection state, and finally to fully deregister and disconnect from CPaaS on application closing.
Once the application code has the Registration Manager, it needs to perform the following tasks:

  1. Call the 'initialize' method to pass in network environment configuration and other options.
  2. Set up an event callback listener by calling 'addConnectionEventListener' and pass in an object containing callbacks to your application code.
  3. Call the 'appStart' method.
  4. If the results of the previous step indicate the system was not previously 'signed-in', the application must call the 'cpaasSignIn' method. This will navigate the browser to the CPaaS service login pages, authenticate the user, allow account line selection, then finally reload the user client.
  • Note: there are three possible states in which a web browser application might start or restart: 1) Fresh clean start with no prior registration or authorization credentials. 2) A restart resulting in the application redirecting away from the application page to the CPaaS Service, which then redirects back to the user application with AuthCode information in the query string parameters. 3) A Browser refresh or restart after the application has signed-in and registered, and still has viable credentials stored in non-volatile memory. In this case, the application does not require the user to sign-in again, and instead can immediately proceed to the operational state.
  1. When the user wants to close the application and clear existing registration credentials, or if they want to login to a different account, they first need to call the 'disconnect' method which will unregister the client from CPaaS.
    If the user just wants to quit the app without signing-out, do not call 'disconnect'. That way when the app starts-up again, it can automatically resume operational status by using cached credentials'.
  2. The Registration Manager has a number of additional methods for querying information associated with registration state. These, and the other methods already mentioned, have JsDocs information.
  3. The CoreSDK may be extended to include accessors to addition common resource interfaces such as event logging.
